# Feature Engineering Agent

> Хакатон **«Код Риска»** · команда **MISIS_STVG**
> Состав: Усенко Тимофей, Крутояров Вячеслав, Пикель Герман, Шубин Вадим

Автономный агент, генерирующий и отбирающий признаки для табличной бинарной классификации. На вход подаётся произвольный реляционный датасет, на выход — до пяти новых признаков, отобранных по реальному вкладу в кросс-валидационный ROC-AUC.

```
┌──────────┐   ┌──────────────┐   ┌─────────┐   ┌───────┐   ┌──────────┐   ┌───────┐   ┌────────┐
│ profiler │ → │ ft_generator │ → │ planner │ → │ coder │ → │ executor │ → │ judge │ → │ writer │
└──────────┘   └──────────────┘   └─────────┘   └───────┘   └──────────┘   └───────┘   └────────┘
                                       ▲             │           │             │
                                       │             └─ retry ───┘             │
                                       └────────────── continue ───────────────┘
```

## Постановка

Хакатонная задача формулируется максимально жёстко: дан произвольный датасет под бинарную классификацию (главная таблица плюс набор справочников), требуется автоматически добавить до пяти признаков, максимизирующих качество CatBoost'а на скрытом тестовом наборе. Главное ограничение — *универсальность*: один и тот же исполняемый pipeline должен корректно отрабатывать на e-commerce, банковском скоринге и телекоме без какой-либо адаптации под конкретную схему данных.

В индустрии задача обычно решается одним из двух полярных способов:

- **Детерминированная автогенерация** (Featuretools, AutoFeat, OpenFE) — exhaustive DFS по реляциям; даёт стабильный, но узкий класс признаков (агрегации, окна, простые отношения).
- **LLM-driven генерация** (CAAFE, AutoML-GPT) — модель пишет код фичей, опираясь на семантику колонок; широкий класс гипотез ценой нестабильности и галлюцинаций.

Реализованное решение объединяет оба подхода: Featuretools обрабатывает рутинные агрегации по графу связей, LLM генерирует семантически осмысленные комбинации, недостижимые для DFS, а отдельная нода-судья пропускает дальше только те признаки, которые статистически значимо двигают метрику.

## Архитектура

Pipeline реализован как направленный граф состояний на LangGraph, состоящий из семи специализированных нод. Разделение ответственности между нодами — ключевое архитектурное решение: оно даёт детерминированную точку отказа, локализует ошибки и позволяет независимо итерировать каждую ступень.

| Нода | Роль |
|------|------|
| `profiler` | Сканирование `data/`, эвристическая идентификация `id_col` и `target_col` по имени и кардинальности, построение профиля каждой таблицы (схема, sample-значения, NaN-доли), расчёт baseline ROC-AUC на сырых колонках |
| `ft_generator` | Построение Featuretools EntitySet с автоматическим определением направления связей parent/child через анализ уникальности ключей; DFS глубины 2 с ограничением `max_features=30` |
| `planner` | Формулировка плана из нескольких feature-гипотез через LLM. В контекст попадают: профиль данных, принятые признаки, отклонённые (последние 10), traceback прошлой ошибки |
| `coder` | Генерация исполняемого pandas-кода под план планировщика. Промпт явно запрещает encoding-операции (one-hot, target, WoE) — CatBoost обрабатывает категориальные признаки нативно |
| `executor` | Исполнение сгенерированного кода в sandbox с `SIGALRM`-таймаутом 60 секунд; валидация инвариантов: row count неизменен, существующие колонки не перезаписаны |
| `judge` | Инкрементальная оценка каждого признака через 5-fold StratifiedKFold; принимается только при приросте ROC-AUC ≥ 0.001 (CAAFE-style гейт) |
| `writer` | Greedy forward selection из принятых кандидатов до финальных пяти по реальному CV-вкладу |

## Ключевые инженерные решения

### Инкрементальный гейт качества

Прямой подход «собрать все сгенерированные признаки и обучить модель» проигрывает на статистически значимую дельту: половина LLM-генераций оказывается шумом, отдельные признаки активно ухудшают модель. Поэтому каждый кандидат проходит индивидуальный 5-fold CV и попадает в пул только при подтверждённом приросте ROC-AUC. Отклонённые признаки и причина отклонения возвращаются в контекст планировщика, что предотвращает повторную генерацию проигравших гипотез — поведение, типичное для одношаговых LLM-пайплайнов.

### Self-correcting prompts

Каждый вызов LLM получает накопленную историю итераций:

- принятые признаки — для исключения дубликатов;
- отклонённые признаки с причиной — для исключения близких вариантов;
- traceback последнего исключения — для исключения повторяющегося подхода;
- развёрнутый план предыдущей итерации — для исключения повторных гипотез.

Эта схема существенно сокращает число «холостых» итераций по сравнению с stateless-промптингом.

### Pre-joined wide-table в executor

Чтобы LLM мог формулировать кросс-табличные агрегации одним выражением (`df.groupby('user_id')['aisle_name'].nunique()`), executor предварительно собирает `order_details` — left-join главной таблицы со всеми справочниками. Альтернативный путь — генерация join-логики на стороне LLM — даёт значительно более высокий процент runtime-ошибок и галлюцинаций имён колонок.

### Адаптивный time budget

Жёсткий потолок исполнения — 600 секунд. На границе каждой ноды сверяется остаток бюджета:

| Остаток | Поведение |
|---------|-----------|
| `< 300с` | Пропускается `ft_generator` (DFS — самая дорогая операция) |
| `< 120с` | Останавливается LLM-цикл, переход к финализации |
| `< 60с` в writer | Fallback на сортировку по `score_delta` вместо greedy CV |

Без этого механизма пайплайн на крупных датасетах гарантированно упирается в таймаут оркестратора.

### Контроль памяти на стадии DFS

Featuretools на справочной таблице в несколько миллионов строк уходит в OOM ещё на построении EntitySet. На входе в `ft_generator` любая таблица, превышающая 500 000 строк, сэмплируется до этого порога — деградация качества генерируемых признаков остаётся в пределах статистического шума, ресурсные затраты сокращаются на порядок.

### Защита от code generation failures

- compile-валидация (`compile(code, '<gen>', 'exec')`) до запуска;
- AST-проверка отсутствия определений функций — допускаются только top-level pandas-выражения;
- автоматический retry до двух раз с передачей traceback в следующий промпт.

## Известные ограничения

- **Стоимость 5-fold под judge'ом.** На крупных датасетах целесообразен адаптивный выбор числа фолдов в зависимости от оставшегося time budget; в текущей реализации это не сделано.
- **Featuretools без custom primitives.** Стандартные примитивы покрывают агрегационный класс, но не достают доменно-специфичных трансформаций. Расширение `featuretools.primitives` под конкретные домены — следующий шаг для повышения качества DFS-части.
- **Жёсткая привязка к одному LLM-провайдеру.** GigaChat подключён через тонкую обёртку без абстракции; вынесение интерфейса в `src/llm/base.py` позволило бы сравнить с альтернативными моделями на той же задаче.
- **Sandbox недостаточен для adversarial-сценариев.** `SIGALRM` плюс AST-фильтр защищают от случайных ошибок, но не от целенаправленно вредоносного кода. Production-grade требует subprocess-изоляции с ограниченной FS.

## Запуск

```bash
uv venv && uv sync
cp .env.example .env       # вписать GIGACHAT_CREDENTIALS
# поместить в data/: train.csv, test.csv, readme.txt + справочники
python run.py
```

Артефакты:

- `output/train.csv`, `output/test.csv` — исходные таблицы с финальными пятью признаками;
- `output/graph/graph_latest.{mmd,png}` — актуальная схема пайплайна.
